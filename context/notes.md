# Notes

- [stack-rationale](notes/stack-rationale.md) — Why Tauri (not web, not TUI), why MDX + KaTeX + Monaco + react-flow, why SQLite WAL, why Claude API.
- [three-pillar-model](notes/three-pillar-model.md) — Teach / Quiz / Interview as branches not steps; `/<topic>` / `/<topic>/quiz` / `/<topic>/interview` routing convention; same content + question bank, three rendering modes.
- [playground-engine-scope](notes/playground-engine-scope.md) — Semi-generic now (`<StepController>` + typed visualisers + per-algorithm code); fully-generic deferred until repetition obviously pays.
- [authoring-discipline](notes/authoring-discipline.md) — Drafts from the sync-learning agent are never auto-applied; widgets stay manual; the component library accretes from real lessons, never speculatively.
- [lessons-as-living-documents](notes/lessons-as-living-documents.md) — Lessons evolve; widget library grows; old lessons get retrofitted with new primitives; backward-compatible widget APIs are required.
- [assessment-design](notes/assessment-design.md) — Memorisation vs understanding tests; 12-shape question taxonomy; `<QuestionGroup>` for progressive difficulty; A-FINE assessment rebuild plan.
- [interface-affordances](notes/interface-affordances.md) — Left-side TOC sidebar; three-tier complexity control (essential/standard/complete) via inline `<Tier>` wrappers; reading progress bar; right-side AI chatbot layout; the `<WidgetFrame>` container + the fullscreen bottom-drawer mini-lesson (distinct from the state caption).
- [llm-integrations](notes/llm-integrations.md) — Four LLM use cases (ask-the-lesson chatbot, conversational interview, graded free-response, sync-learning agent); local-first (Ollama) with Claude API for quality-critical paths; the reader-initiated surfaces — "explain here", ephemeral "turn into lesson", the widget fullscreen mini-lesson.
- [content-architecture](notes/content-architecture.md) — Four-tier content model (full lessons / cross-page hyperlinks / glossary / chat); the concept index that automates the links from authored frontmatter; the hard rule that generation is separated from linking so generated content is cross-linked by construction.
- [visualisation-over-prose](notes/visualisation-over-prose.md) — If a concept depends continuously on a parameter, exhibits a disagreement, or has a silent failure mode, the lesson owes the reader a widget. Prose-only treatment of visualisable concepts is an authoring-discipline failure, not a stylistic choice. Mandatory visualisation audit on every lesson polish pass.
- [lineage-over-snapshot](notes/lineage-over-snapshot.md) — A concept is the surviving answer to a question; teach its genealogy (why it came to be, which alternatives lost and why), its boundary (where it breaks), and its leverage (what it now buys), not just the polished final form. Mandatory lineage audit on every lesson, sibling to the visualisation audit. The `what to teach` companion to visualisation-over-prose's `how to teach`.
- [visual-identity](notes/visual-identity.md) — "Tailored leather notebook meets a configured terminal": warm chocolate-luxe surfaces + terminal-grade mono structure. Signature: the tesseract mark + terminal-pane widgets + boot-cascade motion. ASCII art for custom (non-chart) displays (`src/lib/ascii`). Per-category palettes (identity = structure constant + palette variable). Design values flow from the [styling system](systems/styling-system.md); never hardcode a colour.
- [explanations-must-adapt-to-state](notes/explanations-must-adapt-to-state.md) — Sibling principle. Once a widget is interactive, its caption must be LLM-generated and state-aware via `<WidgetExplainer>`, plus an "Ask a question" affordance. Static "try isolating" fallbacks are a regression. Mandatory state-aware-explanation audit per widget on every polish pass.
- [widget-creativity-discipline](notes/widget-creativity-discipline.md) — The mechanism that keeps widget design from defaulting to "slider + chart." Two-draft rule before building any widget; pedagogical-metaphor library (12 named patterns); acid test for when charts are genuinely the right answer.
- [enrich-lesson-skill](notes/enrich-lesson-skill.md) — Design discussion for a future agent skill that mechanises the lesson audit (visualisation gaps, assessment gaps, cross-page hyperlinks, multi-step goal chains). Not yet built — needs `<GoalChain>` widget designed first.

## Active focus

Past M1, well into M2 (v0.3.3). Shipped since the scaffold: the A-FINE lesson + a 7-lesson linear-algebra track (53 widgets), the Ollama LLM layer, telemetry, the single-source styling system + chocolate-luxe identity, and the ASCII custom-display pass (the rotating donut + ASCII embedding heatmaps; see [`plans/ascii-custom-displays.md`](plans/ascii-custom-displays.md)).

The forward direction is now organised around **two backbones** (see [README §6](../README.md)):

1. **The concept graph (brain)** — a typed DAG from lesson frontmatter feeding navigation, inline concept links, and the links woven into all generated content. The keystone; build first. (→ [`plans/curriculum-graph.md`](plans/curriculum-graph.md))
2. **The token-driven component layer (skin)** — globalised primitives + `<WidgetFrame>` reading one token source, enabling coherence and per-category palettes. (→ [`plans/component-system.md`](plans/component-system.md))

They meet at categories (a category is both a graph root and a colour scheme). On top of the backbones: the fullscreen mini-lesson drawer, "explain here", and ephemeral "turn into lesson" (all generation-separated-from-linking; see [`notes/llm-integrations.md`](notes/llm-integrations.md) §10). Still unbuilt host-side: SQLite spaced-rep, the scheduler, the Quiz/Interview engines, the sync-learning agent, the Claude grader.
